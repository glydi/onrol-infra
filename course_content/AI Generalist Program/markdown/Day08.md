## 2. Completing the n8n Build — The Lead Capture Workflow, End to End
### Picking up the canvas: what the taster build gave you and what comes next
Open your n8n workspace and locate the workflow from Day 7. You should see two nodes on the canvas: the Manual Trigger on the left and the Google Sheets node connected to it on the right. The Manual Trigger is carrying four test data fields: student name, parent contact number, class enrolled, and batch time. The Sheets node is configured to write those fields as a new row in the registration sheet.
The complete workflow has three actions: after the Sheets node writes the row, the workflow needs to send a WhatsApp message to the parent’s contact number confirming the registration, and then publish a notice to the centre’s internal staff group. These two actions will be added in sequence to the right of the Sheets node. In n8n, adding a node is always the same movement: hover over the right edge of the existing node, click the plus button that appears, and select the app and action you want. The data from every previous node is available to every node that follows.
### Adding the WhatsApp confirmation node — mapping data from the trigger into a message
The second action in the workflow is a WhatsApp message sent to the parent’s contact number. In n8n, WhatsApp messages are typically sent through a third-party connector — the most common choices for Indian use cases are Twilio (which supports WhatsApp Business API) or a direct WhatsApp Business node if your account has API access. For the build session today, we are gonna use  Twilio WhatsApp node. Connect your Twilio account credentials in the node configuration panel — the process is a one-time authentication step, the same as connecting any other service.
Once the node is connected, configure the message. The recipient field is the parent’s contact number. In n8n, you reference data from a previous node using the expressions editor — a small panel that opens when you click into any field and shows you all the data available from the nodes that came before. Click into the recipient field, open the expressions editor, and select the contact number field from the Manual Trigger node. It will appear in the field as a reference rather than a fixed value, which means the node will use the actual contact number from the trigger data rather than the same hardcoded number every time.
The message body is written as a text template with field references inserted where the personalised details should appear. A complete message for this workflow looks like this:
Hello! This message confirms that {{studentName}} has been successfully registered at the centre for {{classEnrolled}}, {{batchTime}} batch. We look forward to seeing them. For any questions, please reply to this message.
The double-brace references — {{studentName}}, {{classEnrolled}}, {{batchTime}} — are replaced with the actual values from the trigger when the workflow runs. Every registration produces a message personalised to that student, composed from the same template. Save the node configuration and add a connection from the Sheets node to this new WhatsApp node.
### Adding the staff notification node — Slack or WhatsApp group alert
The third action notifies the centre staff that a new registration has arrived. In many Indian coaching centres, staff communication happens through a WhatsApp group rather than Slack, so both options are covered here — the build follows the same steps whichever you use.
For Slack: add a Slack node after the WhatsApp node, connect your Slack workspace, select the Post a Message action, and choose the channel where the notification should appear. The message body pulls the student name, class, and batch time from the trigger data using the same expression syntax as the WhatsApp node. A complete staff notification message for this workflow is:
New registration: {{studentName}} has enrolled for {{classEnrolled}}, {{batchTime}} batch. Contact number: {{parentContactNumber}}. Record added to the registration sheet.
For a WhatsApp group: the configuration is the same as the parent confirmation node, but the recipient is the group’s registered number rather than the parent’s. The message content is the staff version above rather than the parent version. Whether you use Slack or WhatsApp for the staff notification depends on which platform the centre actually uses — the workflow does not care, only the node configuration changes.
Once this third node is configured and connected, the workflow has its complete linear structure: Manual Trigger → Google Sheets → WhatsApp to parent → Staff notification. Three actions, one path, executing in that order every time the trigger fires.
### Testing the complete workflow: reading the execution log, understanding success and failure
With all three nodes configured, save the workflow and click Execute. Watch the canvas as the execution runs. In n8n, each node lights up as it executes — green for success, red for failure. If all three nodes go green, the workflow has run successfully. Open the Google Sheet and confirm the row is there. Check the WhatsApp test number for the confirmation message. Check the Slack channel or WhatsApp group for the staff notification.
If any node goes red, click on it to open the execution detail. n8n shows you exactly what data entered the node, what the node tried to do, and what error it returned. The most common errors at this stage are authentication failures (the connected account needs to be reauthorised), empty fields in the message template (a reference that did not resolve because the field name was typed incorrectly), and permission errors (the Sheets node does not have write access to the sheet you specified). Each of these is visible in the execution detail and fixable without rebuilding the workflow.
After a successful execution, switch the workflow from the testing Manual Trigger to a real Google Forms trigger. Add a new trigger node — a Google Forms node configured to fire on new form submissions — and remove the Manual Trigger. The rest of the workflow stays exactly as it is. Test with a real form submission. When that runs cleanly, the workflow is live and doing real work.
### What the finished canvas tells you: reading your own workflow after it is built
Step back from the configuration panels and look at the canvas as a whole. You should see four nodes connected in a left-to-right line: the Google Forms trigger, the Sheets node, the WhatsApp node, and the notification node. The connections between them show the direction of data flow. This is the visual representation of the linear framework you classified this workflow as on Day 7 — one trigger, one fixed sequence, one type of result.
The ability to read a finished canvas is as important as the ability to build one. When you hand a workflow over to a client, or return to one you built three months ago, the canvas should tell you immediately what the workflow does without needing to open each node. A canvas that requires you to click into every node to understand the workflow is a canvas that has been built without enough attention to clarity. Names matter: rename each node from its default name to a plain description of what it does — ‘Write to Registration Sheet’, ‘Send Parent Confirmation’, ‘Notify Centre Staff’. A named canvas is readable at a glance. An unnamed one is not.
## 3. Going Deeper into n8n — Control, Data, and Intelligence
### The expressions editor: referencing and transforming data between nodes
You used the expressions editor in Section 2.2 to reference a field from the trigger in the WhatsApp message recipient field. The expressions editor does more than simple field references. It allows you to transform data as it moves between nodes — extracting part of a text string, combining two fields into one, converting a date from one format to another, or running a simple calculation on a number.
The syntax is simple and does not require programming knowledge. A field reference looks like this: {{ $json.fieldName }}. The $json part tells n8n you are referencing data from the current node’s input, and fieldName is the name of the specific field. To reference data from a node other than the immediately previous one, n8n provides a node selector that lets you pick which node’s output you want to pull from.
Transformations are written as extensions of the same expression. To capitalise the first letter of a student name, the expression is {{ $json.studentName.toUpperCase() }}. To combine a student name and class into a single string for a subject line, the expression is {{ $json.studentName + ‘ — ‘ + $json.classEnrolled }}. To extract the first word from a name field (useful if the name is stored as a full name but you want to address the message informally), the expression is {{ $json.studentName.split(‘ ‘)[0] }}.
These transformations are the difference between a workflow that sends ‘Dear Priya Sharma’ in a message body and one that sends ‘Dear Priya’. They make outputs feel genuinely personalised rather than mechanically assembled. Learning to use the expressions editor well is one of the highest-return skills in n8n, because it removes the need to pre-format data in Google Sheets or in a separate processing step.
### The router node: turning the linear build into a data-driven one
The workflow you just built is linear: every registration produces the same three actions regardless of what the data contains. A real centre often needs the workflow to behave differently in certain situations. Consider a batch that is full: when the morning batch reaches its maximum enrolment, new registrations for that batch should be redirected to a waitlist sheet and the parent should receive a different message explaining the situation.
This is a data-driven requirement, and the tool that implements it in n8n is the router node — sometimes called the IF node or Switch node depending on the version. Add a router node between the Sheets node and the WhatsApp node. Configure it with a condition: if the batch time field equals ‘Morning’ and the current morning batch count (read from a separate tracking cell in the sheet) is greater than the maximum allowed, route to path B. Otherwise, route to path A.
Path A is the original workflow: the normal parent confirmation and the staff notification. Path B is a new branch: a write to the waitlist sheet, a different WhatsApp message to the parent explaining the waitlist, and a staff notification flagging the overflow. The canvas now shows two paths leaving the router node, each leading to its own set of action nodes.
This is precisely the transition from the linear framework to the data-driven framework that Day 7 described. The trigger and the Sheets node are shared by both paths. The router is the decision point. Everything after the router differs by path. The skill of adding this branch to an existing linear workflow — without rebuilding from scratch — is what distinguishes someone who can maintain and extend automation systems from someone who can only follow a tutorial to build them once.
### The AI node: connecting ChatGPT inside a workflow
n8n has a built-in AI node that connects directly to OpenAI’s API. This node accepts a text input, sends it to a language model with a prompt you define, and returns the model’s response as an output that subsequent nodes can use. The connection between Module 1’s prompt engineering and Module 2’s automation is most visible here: the same discipline of writing a clear, well-structured prompt applies inside a workflow node exactly as it does in a chat interface.
A practical use of the AI node in the lead capture context is message personalisation. Instead of a fixed WhatsApp template, the workflow passes the student’s name, class, and batch time to the AI node with a prompt that asks it to write a warm, personalised confirmation message in simple Indian English. The AI node returns the message, and the WhatsApp node sends it. Every parent receives a message that reads as though someone sat down and wrote it for their specific child, produced automatically for every registration.
A second use is classification. Suppose the registration form has an open-text field where parents can note any special requirements. Rather than a human reading each entry and deciding whether it needs urgent attention, the AI node receives the text and a prompt:
Read the following parent note and classify it as either URGENT (requires a staff call before the first session), STANDARD (can be handled at first session), or NONE (no special requirement noted). Reply with only one word: URGENT, STANDARD, or NONE. Parent note: {{specialRequirements}}
The AI node’s single-word response becomes the input to a router node. URGENT routes to a node that creates a task in the centre’s task manager and sends a staff alert. STANDARD routes to a note added to the registration sheet. NONE continues through the standard workflow without any additional action. The AI is making a judgement call inside the workflow — the same judgement a person would make when reading the note — and the classification becomes the data that the data-driven framework routes on. This is the AI-augmented condition pattern from Day 7, running live.
### Error handling in n8n: what happens when a node fails and how to manage it
Every workflow that runs on real data will eventually encounter an error. A WhatsApp number is missing. The Google Sheet is at its row limit. Twilio is temporarily unavailable. The AI node returns an unexpected response. How a workflow handles these failures is the difference between a system that a client can trust and one that silently breaks and leaves them with gaps in their data.
n8n has two mechanisms for managing errors. The first is the error output of a node. Every node in n8n has a standard output (the green path, carrying data when the node succeeds) and an error output (the red path, carrying error information when the node fails). By default, the error output is not connected to anything, which means failures are logged but the workflow stops. You can connect the error output to a separate set of nodes — for example, a Slack notification to the operator that a node failed, combined with a note written to an error log in Google Sheets. This way, a failure does not go unnoticed.
The second mechanism is the Continue on Error setting inside individual nodes. When this is enabled, a node that fails does not stop the workflow — it passes the error information to the next node and lets the workflow continue. This is appropriate for non-critical actions: if the staff notification fails, the registration record and the parent confirmation are still more important. With Continue on Error enabled on the notification node, a temporary Slack outage does not cause the whole workflow to fail.
A well-designed workflow handles its most important action first and makes its least critical actions tolerant of failure. For the lead capture workflow: writing the registration record is the most important action and should stop the workflow if it fails, because a missing record is a serious problem. The parent confirmation is the next most important. The staff notification can be set to continue on error. Thinking through this priority order during the design phase — not after the first production failure — is what the fifth question in Day 6’s five-question design framework was asking you to do.
### Why n8n is the professional-grade choice: self-hosting, data control, and scale
The features you have just worked through — the expressions editor, the router, the AI node, and structured error handling — are available in n8n and not fully available in the other two platforms at the same level of control. But the most important distinguishing feature of n8n is not any one of these. It is the ability to run n8n on your own infrastructure.
When you build a workflow in Zapier or Make, your data passes through their cloud servers. For most personal and small-business automations, this is perfectly acceptable. But many business clients — particularly in finance, healthcare, legal services, and government-adjacent work — cannot allow their data to be processed by a third-party cloud service they have not vetted. A coaching centre’s student details are relatively low-sensitivity. On the other hand, a hospital’s patient records, a chartered accountant firm’s client financial data, or an HR team’s salary information is not.
Self-hosted n8n runs on a server that the client controls. Their data never leaves their own infrastructure. The automation platform is theirs. Updates are applied on their schedule. This is what opens the door to enterprise and institutional clients — clients who cannot use Zapier or Make but can use a self-hosted n8n installation. Installing and configuring self-hosted n8n is within reach of anyone who can follow documented steps; it does not require a software engineering background. And the ability to offer it separates you, as a freelancer, from the large majority of automation practitioners who work only in the cloud platforms.
## 4. Zapier in Practice — Building the Same Workflow as a Zap
### Setting up the Zap: choosing the trigger app and event
Open Zapier and click Create Zap. The builder presents a vertical list of steps, starting with the trigger. Click on the trigger step and search for Google Forms. Select the trigger event: New Form Response. Connect your Google account when prompted and select the specific form you want to use. Zapier will pull in a sample response from the form so you can see the fields available. This sample is what you will map into the action steps.
This is the clearest structural difference from n8n immediately visible: in n8n, you placed nodes on a canvas and connected them with lines. In Zapier, you are filling in a vertical form. There is no canvas. The structure of the workflow is implied by the order of the steps rather than drawn explicitly. For someone building their first automation, this is easier to follow. For someone who needs to see the full shape of a complex workflow at a glance, it is a limitation.
### Adding action steps and mapping data in Zapier's step-by-step builder
Below the trigger step, click the plus button to add the first action. Search for Google Sheets and select the action: Create Spreadsheet Row. Connect your Google account, select the spreadsheet and the specific sheet within it, and map the columns. Zapier shows you each column header from your sheet and lets you click into the corresponding field to select the data from the form response. The mapping interface is visual and guided — it shows you available fields from the trigger as a dropdown, and you click to insert them.
Add the second action: search for WhatsApp or Twilio and select Send WhatsApp Message. Configure the recipient number by clicking into the field and selecting the phone number field from the form response. Write the message body with field insertions the same way you mapped the Sheets columns — clicking into the text field and inserting form response fields where the personalised content should appear.
Add the third action: search for Slack and select Send Channel Message. Choose the channel, write the staff notification message, and insert the relevant fields. By this point the Zap has its three action steps, in the same order as the n8n workflow: Sheets write, WhatsApp to parent, staff notification.
### Testing and publishing the Zap — what the task log shows you
Zapier tests each step as you build it. After configuring each action, there is a Test button that runs that step with the sample data from the trigger and shows you the result. If the step succeeds, Zapier shows a green tick and displays the output. If it fails, it shows the error message. This step-by-step testing is useful for catching configuration errors early, before the whole Zap is live.
Once all three action steps are tested and confirmed, click Publish. The Zap is now live and will run every time a new Google Forms response arrives. Zapier’s task history — accessible from the left sidebar under Tasks — shows every run: when it fired, which steps succeeded, and how many tasks were consumed. On the free plan, remember that each action step counts as one task, so the three-action Zap consumes three tasks per form submission.
### Where Zapier wins and where it reaches its limits
Building the same workflow in Zapier that you just built in n8n reveals the trade-off clearly. The Zapier build was faster — the step-by-step interface guided you through each decision without requiring you to place and configure nodes manually. The app library is enormous, so the Google Forms, Google Sheets, Twilio, and Slack connectors were all immediately available. For a straightforward linear workflow, Zapier is genuinely the fastest path from idea to live automation.
The limits appear the moment the workflow needs to do something the step-by-step builder does not support directly. Adding a condition — the batch-full router from Section 3.2 — in Zapier requires a Filter step and then a separate Zap for the alternate path, because Zapier does not have a visual branching canvas. Building the AI node equivalent requires a Zapier AI step or a Webhooks step that calls the OpenAI API directly, neither of which is as clean as n8n’s dedicated AI node. And self-hosting is not an option; your data flows through Zapier’s servers regardless.
The right mental model for Zapier is this: it is the fastest tool for simple, linear, common-app workflows that do not need fine-grained control. When a client’s entire automation need is a two-step Zap between two widely used apps, Zapier is the right answer. When the requirement is more complex, the speed advantage disappears and the control disadvantage becomes the dominant factor.
## 5. Make in Practice — Building the Same Workflow as a Scenario
### The Make canvas: modules, bundles, and the visual flow
Open Make and create a new scenario. The canvas is immediately familiar from n8n — a visual workspace where you place and connect components — but Make’s visual language is different. Each step in a Make scenario is a circle called a module, and connections between modules are drawn as lines with arrows. The first module is always the trigger, placed at the top of the canvas, with action modules connected to it in sequence.
Where n8n passes data between nodes as JSON objects with named fields, Make passes data as bundles — packets of information that flow from one module to the next. The data available in any module is everything that has been produced by the modules before it in the sequence. Make’s data mapping interface displays the available bundle contents as a list that you drag or click into the fields of the current module. The feel is slightly different from n8n’s expressions editor, but the underlying concept — data from earlier steps is available to later steps — is identical.
### Building the scenario: trigger module to action modules
Add the first module: search for Google Forms and select Watch Responses. Connect your account, select the form, and save. Make will ask you to run the scenario once to pull in a sample response — do this by submitting a test response to the form. The sample bundle now contains all the form fields.
Add the second module connected to the first: Google Sheets — Add a Row. Select your spreadsheet and sheet, then map each column from the bundle fields. Add the third module: Twilio — Send WhatsApp Message. Map the recipient number and compose the message body using bundle fields. Add the fourth module: Slack — Create a Message. Select the channel and compose the staff notification. The scenario now mirrors the n8n and Zapier builds exactly — the same workflow, on a visual canvas, in Make’s circular module style.
Click Run Once to test the scenario with real data. Make shows each module’s execution in the canvas, with the number of bundles processed displayed inside each circle. A green circle means the module succeeded; a red one means it failed. The execution detail for each module is accessible by clicking on it — Make shows you the input bundle and the output bundle, which makes it straightforward to see exactly what data went into each step and what came out.
### Where Make fits relative to n8n and Zapier
Building the same workflow in Make reveals where it sits in the landscape. The visual canvas makes the shape of the scenario immediately clear in a way that Zapier’s vertical list does not. The data mapping interface is more visual than n8n’s expression editor and slightly easier for someone who finds syntax unfamiliar. Make’s data processing tools — built-in modules for filtering records, aggregating values, and transforming text — are more accessible than Zapier’s equivalents and cover most of the data manipulation that n8n handles with expressions.
The limits are the same ones identified conceptually on Day 6: Make’s free plan caps at one thousand operations per month with two active scenarios, its data passes through Make’s cloud servers, and its AI capabilities are less integrated than n8n’s dedicated AI nodes. Make is the right choice when the workflow is moderately complex, when a visual canvas matters for readability, and when the client’s data does not require the control that self-hosting provides. For straightforward client work with common apps, it is a strong and cost-effective option.
## 6. Choosing Your Platform — A Working Decision Framework
### The three deciding factors: workflow complexity, data sensitivity, and who maintains it
Having built the same workflow three times, the decision between platforms can now be grounded in direct experience rather than abstract description. Three factors determine the right choice for any given job.
The first is workflow complexity. A workflow with two or three actions, no branching, and common apps is equally achievable in all three platforms. Zapier is the fastest to build. As complexity increases — multiple branches, data transformations, AI steps, error handling, sub-workflows — n8n’s control advantages become material and the other two platforms start to show their ceilings. A useful rule: If your workflow follows a single, linear sequence of steps, most automation platforms will meet your needs. However, if it involves conditional logic, branching paths, or multiple possible outcomes, n8n is the more reliable choice.
The second is data sensitivity. If the workflow touches information that a client would not want stored on a third-party server — patient data, financial records, confidential business communications — n8n self-hosted is the only option of the three. Zapier and Make are cloud services; that is not a flaw, but it is a constraint that some clients will not accept.
The third is who will maintain the workflow after the build. A workflow handed to a non-technical client to manage themselves is best built in Zapier, because its interface is the most approachable for someone who did not build the workflow. A workflow that you will maintain as part of a retainer can be built in whichever platform you are most productive in. A workflow being handed to a client’s technical team is best built in n8n, because it is the platform that technical teams are most likely to already use or be able to extend.
### The platform comparison at a glance — and when to use more than one
A brief summary of what today’s builds confirm about each platform: Zapier is the fastest to learn and use for simple linear workflows, has the largest app library, costs the most at volume, and offers no self-hosting option. Make is visually clear, handles moderate data processing well, is more affordable than Zapier at equivalent volumes, and also requires cloud processing. n8n offers the highest control, the most powerful data handling, the best AI integration, a free self-hosted option, and a steeper initial learning curve that today’s session has largely addressed.
It is also worth knowing that using more than one platform is normal and often sensible. A freelancer might build simple client-facing Zaps in Zapier (because the client can see and understand the Zap history themselves) while running more complex internal workflows on a self-hosted n8n instance. A larger client organisation might use Zapier for its marketing team’s lightweight automations and n8n for its operational workflows that handle sensitive customer data. Knowing all three platforms means you can advise on the right tool rather than defaulting to the only one you know.
## 7. Turning Automation Tool Skills Into Income
### What clients pay for at the tool level — not just “automation” but specific platform expertise
On Day 6, the monetisation discussion focused on the types of automation services that Indian businesses buy: lead capture, invoice reminders, internal notifications, report automation. Today the conversation moves one level more specific: within those service categories, clients increasingly ask which platform you work in, and your answer affects both the clients you can reach and the prices you can charge.
The reason is that businesses do not buy ‘automation’ in the abstract. They buy a lead capture system that runs in n8n, or a Zap that connects their Typeform to their CRM(Content Resource Management) , or a Make scenario that processes their daily inventory report. When a client already has some automations in one platform and wants more, they need a practitioner who knows that platform specifically, because their existing workflows need to be understood and extended rather than rebuilt from scratch. Platform-specific expertise is therefore a genuine differentiator — not a superficial one — and the platform you specialise in determines the type of client who can use you.

### Packaging a platform-specific offering and pricing it
A platform-specific offering is more specific and more compelling than a general automation offer. Compare these two pitches. The first: ‘I build automations for small businesses.’ The second: ‘I build lead capture and notification systems in n8n for coaching centres and real-estate teams, including a self-hosted option for clients who need their data to stay on their own server.’ The second pitch names the platform, the workflow types, and the specific client segment, and it mentions a capability — self-hosting — that most of their current practitioners cannot offer. A coaching centre director who reads the second pitch knows immediately whether it applies to them.
For pricing, today’s builds give you direct reference points. The three-action lead capture workflow you built today is a linear workflow in the Day 7 pricing framework: two thousand to five thousand rupees as a one-time build. The version with the router node added — the data-driven extension from Section 3.2 — moves into the moderate complexity range: eight thousand to fifteen thousand rupees. The version with the AI node for message personalisation and classification adds another layer of design and testing, pushing the top of that range. A self-hosted n8n installation for a client — server setup, n8n installation, workflow migration, documentation — is a separate service that sits above the workflow pricing and is typically quoted as a fixed project fee.
The pattern to carry out of this section is the same one from Day 6 and Day 7: start with one specific offering, build it for one real client or produce two or three samples for a client you know, and let the quality of the work produce the next conversation. A coaching centre manager who sees a polished, working lead capture system — a system that handled fifty new term registrations without a single manual entry — is the most effective sales material for the next coaching centre you approach.
